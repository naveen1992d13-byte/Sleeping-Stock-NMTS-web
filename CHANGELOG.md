# CHANGELOG — Parts Transfer Request: Number, PDF, Gmail Email

## Files changed
- `backend/server.py` — request numbering, grouping, idempotency, indexes, new endpoints
- `backend/notifications.py` — PDF generation, Gmail email w/ attachment, shared SMTP config
- `backend/requirements.txt` — added `reportlab==4.4.10`
- `backend/.env.example` — documented SMTP_* fallback names (no new required vars)
- `frontend/Orders.js` — Send Request UX (loading state, request number + email status banner, resend)
- `frontend/Requests.js` — Request Number + LOC columns, safe fallback for legacy rows

`App.js` was reviewed — no changes needed; existing routes/permissions already cover Request Center.
`PrintStyles.js` / `printLayout.js` was not part of the uploaded files, so it was not touched.

## 1. Request number
`RQ{BRAND}{YYMMDD}{0001-9999}` e.g. `RQHY2607150001`.
- Brand code resolved via the existing `resolve_brand_code_for_upload()` (Brand Master lookup,
  never hardcoded).
- Generated by `_generate_request_number()`: an atomic `$inc` on a per-brand-per-day counter
  document (`db.counters`, id `request_RQ_{BRAND}_{YYMMDD}`) — the same pattern already used for
  order numbers, so two simultaneous requests can never collide, and the serial naturally resets
  to `0001` for a new brand or date.
- Used consistently on: the request_headers group document, every order_requests item row,
  Request Center list/detail, the PDF, the email subject, the attachment filename, and search.

## 2. Send Request workflow (`POST /order-desk/orders/{id}/send-requests`)
Rewritten to group allocations by **Requested-To destination** (supplying dealer + branch):
- One `request_headers` document (one request number, one PDF, one email) per destination.
- The existing per-item `order_requests` rows are still created — one per (item, source) — so
  Approve/Reject/Cancel and the Request Center list work exactly as before; each row now also
  carries `request_number` and `request_group_id`.
- Email is sent only after the request/items are saved, and a PDF/SMTP failure only updates
  `email_status`/`email_error` — it never rolls back the saved request.
- Response includes `request_created`, `request_numbers`, `email_sent`, `email_error`, `message`.

## 3. Duplicate prevention / idempotency
Two independent guards:
1. **Content fingerprint** — SHA-256 of the order's current item/allocation content
   (`_send_requests_fingerprint`, order-independent of item ordering). Stored on the order as
   `send_requests_fingerprint`. A repeated click/retry with identical content returns the
   previously created groups instead of creating new ones.
2. **Unique index** — `request_headers` has a unique index on
   `(order_id, supplying_dealer, supplying_branch)` plus a unique index on `request_number`. A rare
   concurrent race is caught as `DuplicateKeyError`, the just-inserted item rows for that race are
   rolled back, and the winning group is returned instead.

New endpoint `POST /requests/group/{request_number}/resend-email` — a safe, explicit retry for a
request whose email previously failed; never creates a new request or request number, and is a
no-op if the email already sent successfully.

## 4. PDF (ReportLab, `notifications.build_request_pdf`)
- Title "Sleeping Stock / Parts Transfer Request", Request Number, Order Number, Created date,
  Status, Requested By / Requested To blocks, item table (S.No, Part Number, Description, Req.
  Qty, Avail. Qty, Value, Purchase Aging, Sales Aging, LOC), totals, signature block.
- Multi-page: item table header repeats on every page (`repeatRows=1`); signature block is the
  last flowable so it only ever renders on the final page; "Page X of Y" via a two-pass
  `NumberedCanvas`.
- Backward compatible: missing `purchase_aging_days` / `sales_aging_days` / `loc` render as `—`
  instead of crashing.

## 5. Aging & LOC rule
Purchase Aging, Sales Aging, and LOC are taken from the **source/supplying stock at the moment of
allocation** (added to the availability-check and allocate responses as `loc`, alongside the
already-existing `purchase_aging_days` / `sales_aging_days`), never from the requester's branch or
a shared aging field. These are frozen onto each `order_requests` row as `purchase_aging_days_at_request`,
`sales_aging_days_at_request`, `loc_at_request` at request-creation time, and copied onto the PDF's
line items — so later stock changes never retroactively alter a past request's PDF.

## 6. Multiple source branches
Each distinct (supplying dealer, supplying branch) destination gets its own request number, its own
`request_headers` document, its own PDF, and its own email — never combined into one PDF for
unrelated branches.

## 7. Receiver email resolution (`_resolve_request_receiver_email`)
Exact-match active Admin at the supplying dealer+branch; falls back to an active Master for the
supplying brand; returns `""` (never guesses) if neither exists. If empty/invalid,
`email_status` is set to `failed` with `"Receiver email not configured"` and the request itself
still succeeds.

## 8. Gmail SMTP
`notifications.gmail_settings()` now resolves either naming style:
`GMAIL_SMTP_HOST/PORT/USERNAME/APP_PASSWORD/SENDER_NAME` or `SMTP_HOST/PORT/EMAIL/PASSWORD/FROM_NAME`
(GMAIL_SMTP_* takes priority when both are set). Used by both the existing approve/reject email
path and the new Parts Transfer Request email path. Credentials are never logged; SMTP errors are
scrubbed of the password before being stored/returned.

## 9. Email format
Subject: `Parts Transfer Request - {REQUEST_NUMBER}`. Attachment: `{REQUEST_NUMBER}.pdf`. HTML
body with a plain-text fallback; body is a short summary only (no item table — that's in the PDF).

## 10. Notification logging
`notification_logs` extended with: `request_number`, `attachment_filename`, `retry_count`,
`provider_message_id`, alongside the existing `request_id`/`order_id`/`receiver_email`/`status`/
`error_message`/timestamps. New indexes on `request_number`, `receiver_email`, `status`, `created_at`.

## 11–12. Backend structure / PDF library
New helpers added to `notifications.py`: `build_request_pdf`, `send_request_pdf_email`,
`gmail_settings`. New helpers added to `server.py`: `_generate_request_number`,
`_send_requests_fingerprint`, `_resolve_request_receiver_email`, `_send_request_group_email`,
`_create_request_group`, `_send_requests_response`. `reportlab` chosen over WeasyPrint/xhtml2pdf —
pure-Python, no OS-level dependencies, simplest deployment.

## 13. Frontend
**Orders.js** — dedicated `sendingRequest` state disables the button and shows "Sending
Request..."; a result banner shows the generated request number(s) and email success/failure
*separately* (a failed email is never presented as a failed request); a "Resend Email" button
appears per request number when the email failed, calling the new resend endpoint.
**Requests.js** — added "Request No" and "LOC" columns (both render a safe fallback for legacy
rows that predate these fields); Purchase/Sales Aging columns already had safe `?? '-'` fallbacks.
Existing Approve/Reject/Cancel/Print flows untouched.
**App.js** — reviewed, no changes required.

## 14–16. Database compatibility / security / failure handling
All new fields are additive; nothing is migrated or deleted. Older `order_requests` rows without
`request_number`/`loc_at_request` still load and print (shown as `— (legacy)` / `-`). Recipients
and PDF content are always resolved/generated server-side — the frontend never supplies or
receives raw email targets or PDF bytes for this flow.

## Verification performed in this sandbox
This sandbox has no network access and does not have `motor`, `pymongo`, `fastapi`, or `pydantic`
installed, so a live server boot / real MongoDB / real Gmail send could not be executed here.
What *was* verified:

- **Python syntax**: `python3 -m py_compile server.py notifications.py` — pass, no errors.
- **Frontend syntax**: `tsc --noEmit --jsx react` against `App.js`/`Orders.js`/`Requests.js` (JSX
  parse-level check) — pass, no errors, both before and after the edits.
- **Diff review**: confirmed the only removed lines are the old send-requests implementation being
  replaced; nothing else was altered.
- **Request-number logic** (17 tests against the *exact* extracted functions from `server.py`,
  run against an in-memory fake Mongo-like collection replicating `$inc`/upsert/unique-index
  semantics): format, no hyphens/spaces, brand code from Brand Master (not hardcoded), 50
  concurrent calls for the same brand/day produce 50 unique sequential numbers, a different brand
  restarts at 1, fingerprint is stable regardless of item ordering and changes when content
  changes, receiver-email resolution (exact match → brand fallback → empty), and the unique index
  rejects a duplicate destination group. **All 17 passed.**
- **PDF generation**: a 60-item request rendered to a correct 6-page PDF (verified visually) —
  header/branding on page 1, table header repeated on every page, totals + signature block only
  on the final page, correct "Page X of Y". A second PDF with `None`/`""` aging and LOC fields
  (simulating legacy records) rendered without error, showing `—` instead of crashing. An
  empty-items PDF also rendered without error.
- **Email failure paths**: invalid email → `skipped`/`invalid_or_missing_email`; SMTP not
  configured → `skipped`/`gmail_not_configured`; unreachable SMTP → `failed` with the error message
  and confirmed the App Password never appears in it; legacy `SMTP_EMAIL`/`SMTP_PASSWORD` env vars
  resolve correctly through `gmail_settings()`.

## What could not be tested here
- Booting the actual FastAPI app against a real MongoDB (packages/network unavailable in this
  sandbox) — please run `pip install -r requirements.txt` and start the server in your own
  environment; `ensure_product_hub_indexes()` will create the new indexes automatically on startup.
- An actual Gmail send (needs a real App Password and outbound network).
- `craco build` — the full CRA/CRACO project (package.json, node_modules, craco.config.js) wasn't
  part of the uploaded files, only the individual components; a JSX-level parse check was
  substituted (see above). Please run `yarn build` / `npm run build` in your project to do a full

- `PrintStyles.js` / `printLayout.js` wasn't included in the upload, so its `PrintStyles`/`PrintHeader`
  exports (used unchanged by `Requests.js`) could not be inspected — if its print CSS assumes a
  fixed column count, you may want to double check the new "Request No"/"LOC" columns look right
  in the browser print view.
